# PRD: 기업 소개 플랫폼 (Company Discovery Platform)

> 작성일: 2026-05-22
> 상태: Ready for Implementation

---

## Problem Statement

기업들은 자신의 정보를 잠재 고객, 파트너, 구직자 등에게 효과적으로 소개할 수 있는 전용 공간이 부족하다. 기존 SNS나 포털은 기업 소개에 최적화되어 있지 않고, 방문자 입장에서도 원하는 기업을 체계적으로 탐색하기 어렵다. 또한 자연어로 기업을 검색하거나 특정 기업에 대해 질문할 수 있는 인터페이스가 없다.

---

## Solution

기업이 직접 회원가입하여 자사 프로필 페이지를 작성하고, 방문자는 로그인 없이 자유롭게 기업 정보를 열람할 수 있는 웹 플랫폼을 구축한다. AI 챗봇(Ollama + RAG)을 통해 자연어로 기업을 검색하거나 특정 기업에 대해 질문할 수 있다. 관리자는 기업 가입 승인/거절 및 프로필 관리 권한을 가지며, 검증된 기업만 플랫폼에 노출된다.

---

## User Stories

### 방문자 (비로그인)

1. 방문자로서, 로그인 없이 기업 목록을 열람할 수 있어서 진입 장벽 없이 원하는 기업 정보를 찾을 수 있다.
2. 방문자로서, 기업명으로 검색할 수 있어서 원하는 특정 기업을 빠르게 찾을 수 있다.
3. 방문자로서, 업종별로 기업을 필터링할 수 있어서 관심 있는 분야의 기업만 볼 수 있다.
4. 방문자로서, 지역별로 기업을 필터링할 수 있어서 특정 지역의 기업만 볼 수 있다.
5. 방문자로서, 업종 카테고리 메뉴를 통해 탐색할 수 있어서 직접 검색어를 입력하지 않아도 기업을 발견할 수 있다.
6. 방문자로서, 기업 프로필 페이지에서 회사명, 로고, 소개, 업종, 설립연도, 직원 수, 규모, 웹사이트, 연락처, 주소, 이미지 갤러리, SNS 링크를 확인할 수 있어서 해당 기업을 충분히 파악할 수 있다.
7. 방문자로서, AI 챗봇에게 자연어로 기업을 검색할 수 있어서 ("서울에 있는 IT 스타트업 알려줘") 직관적으로 원하는 기업을 찾을 수 있다.
8. 방문자로서, AI 챗봇에게 특정 기업에 대해 질문할 수 있어서 ("OO기업은 어떤 회사야?") 프로필 기반의 정확한 답변을 얻을 수 있다.
9. 방문자로서, 기업 목록에서 카드 형태로 기업을 미리 볼 수 있어서 클릭 전에 기본 정보를 파악할 수 있다.
10. 방문자로서, 기업 프로필의 웹사이트 링크를 클릭할 수 있어서 해당 기업의 공식 사이트로 이동할 수 있다.
11. 방문자로서, 기업의 SNS 링크(인스타그램, 링크드인 등)를 확인할 수 있어서 다양한 채널에서 기업을 팔로우할 수 있다.
12. 방문자로서, 기업 이미지 갤러리를 볼 수 있어서 기업의 분위기나 제품을 시각적으로 파악할 수 있다.

### 기업 회원

13. 기업 담당자로서, 이메일과 비밀번호로 회원가입할 수 있어서 플랫폼에 자사 프로필을 등록할 수 있다.
14. 기업 담당자로서, 가입 시 이메일 인증을 거쳐서 계정의 신뢰성을 증명할 수 있다.
15. 기업 담당자로서, 로그인 후 자사 프로필을 작성할 수 있어서 방문자에게 보여줄 정보를 등록할 수 있다.
16. 기업 담당자로서, 프로필에 회사명, 로고, 소개 텍스트, 업종, 설립연도, 직원 수, 규모를 입력할 수 있어서 기업의 기본 정보를 충분히 표현할 수 있다.
17. 기업 담당자로서, 프로필에 웹사이트 링크, 이메일, 전화번호, 주소를 입력할 수 있어서 관심 있는 방문자가 연락할 수 있다.
18. 기업 담당자로서, 프로필에 이미지를 여러 장 업로드할 수 있어서 기업의 모습을 시각적으로 보여줄 수 있다.
19. 기업 담당자로서, SNS 링크(인스타그램, 링크드인 등)를 등록할 수 있어서 방문자가 다양한 채널에서 기업을 팔로우할 수 있다.
20. 기업 담당자로서, 작성한 프로필이 관리자 승인 후 공개됨을 안내받아서 승인 절차를 이해할 수 있다.
21. 기업 담당자로서, 승인 상태(대기중/승인/거절)를 확인할 수 있어서 현재 프로필 상태를 파악할 수 있다.
22. 기업 담당자로서, 승인 거절 시 거절 사유를 확인할 수 있어서 프로필을 수정하여 재신청할 수 있다.
23. 기업 담당자로서, 이미 승인된 프로필을 수정할 수 있어서 최신 정보를 유지할 수 있다.
24. 기업 담당자로서, 프로필을 수정하면 재승인 절차를 거쳐서 플랫폼의 콘텐츠 신뢰성이 유지된다.
25. 기업 담당자로서, 비밀번호를 재설정할 수 있어서 비밀번호를 분실해도 계정에 다시 접근할 수 있다.

### 관리자

26. 관리자로서, 별도의 관리자 계정으로 로그인할 수 있어서 일반 기업 계정과 분리된 관리 권한을 가진다.
27. 관리자로서, 승인 대기 중인 기업 프로필 목록을 볼 수 있어서 검토가 필요한 항목을 파악할 수 있다.
28. 관리자로서, 기업 프로필을 상세 조회하고 승인 또는 거절할 수 있어서 플랫폼에 등록되는 기업을 검증할 수 있다.
29. 관리자로서, 거절 시 사유를 입력할 수 있어서 기업 담당자가 수정 방향을 알 수 있다.
30. 관리자로서, 이미 승인된 기업 프로필을 삭제할 수 있어서 부적절한 콘텐츠를 제거할 수 있다.
31. 관리자로서, 전체 기업 목록을 조회하고 상태(승인/대기/거절)별로 필터링할 수 있어서 플랫폼 현황을 파악할 수 있다.

## Implementation Decisions

### 아키텍처

- **프론트엔드:** Nuxt.js (Vue 3) + Tailwind CSS — SSR/SSG 지원으로 SEO 유리, 기업 프로필 페이지의 검색 노출에 중요
- **백엔드:** FastAPI (Python) — 비동기 처리, Ollama 연동 용이, Pydantic 기반 스키마 검증
- **데이터베이스:** PostgreSQL (로컬 Docker) — 프로덕션 배포 시 그대로 사용 가능
- **AI:** Ollama (llama3.1:8b) + RAG — 완전 무료 로컬 LLM, 기업 프로필 데이터를 컨텍스트로 주입

### 주요 모듈

#### 백엔드 (FastAPI)

1. **Auth 모듈** — JWT 발급/검증, 이메일 인증 토큰 생성/검증, 비밀번호 해싱 (bcrypt)
2. **Company 모듈** — 기업 프로필 CRUD, 승인 상태 관리, 이미지 업로드 처리
3. **Admin 모듈** — 승인/거절 API, 기업 목록 관리
4. **Search 모듈** — 키워드 검색, 업종/지역 필터, PostgreSQL Full-Text Search 활용
5. **AI Chat 모듈** — Ollama API 연동, RAG 파이프라인 (질문 → DB 검색 → 컨텍스트 주입 → 응답 생성)

#### 프론트엔드 (Nuxt.js)

1. **Auth Store** — 기업 로그인/로그아웃 상태 관리 (Pinia)
2. **Company Profile 페이지** — 기업 프로필 상세 조회
3. **Company List 페이지** — 기업 목록, 검색/필터 UI
4. **Dashboard 페이지** — 기업 회원용 프로필 관리
5. **Admin 페이지** — 관리자 승인/거절 UI
6. **AI Chat 컴포넌트** — 챗봇 인터페이스 (플로팅 버튼 또는 사이드바)

### 데이터 스키마 (핵심)

```
Company {
  id, email, password_hash, status (pending/approved/rejected),
  name, logo_url, description, industry, founded_year,
  employee_count, company_size, website, contact_email,
  phone, address, image_urls[], instagram_url, linkedin_url,
  rejection_reason, created_at, updated_at
}

Admin {
  id, email, password_hash
}
```

### RAG 파이프라인

1. 사용자 질문 수신
2. PostgreSQL에서 관련 기업 검색 (Full-Text Search + 키워드 매칭)
3. 검색된 기업 프로필 데이터를 프롬프트 컨텍스트로 구성
4. Ollama llama3.1:8b에 컨텍스트 + 질문 전달
5. 생성된 답변 반환

### 인증 흐름

- 기업 가입 → 이메일 인증 → 프로필 작성 → 관리자 승인 대기 → 승인 시 공개
- JWT Access Token (단기) + Refresh Token (장기) 방식
- 관리자 계정은 별도 role 필드로 구분

### 이미지 저장

- FastAPI `StaticFiles`로 서빙
- 파일명은 UUID로 생성하여 충돌 방지
- 추후 Cloudinary 또는 S3로 교체 시 URL만 변경

## Testing Decisions

### 좋은 테스트의 기준

- 내부 구현이 아닌 외부 동작(API 응답, UI 상태 변화)을 테스트
- 각 테스트는 독립적으로 실행 가능해야 함
- 실패 시 원인을 명확하게 파악할 수 있어야 함

### 테스트 대상 모듈

| 모듈 | 테스트 종류 | 우선순위 |
|------|------------|---------|
| Auth API | 단위 + 통합 | 높음 |
| Company CRUD API | 통합 | 높음 |
| Admin 승인/거절 API | 통합 | 높음 |
| Search API | 통합 | 중간 |
| AI Chat API | 통합 (Ollama Mock) | 중간 |
| Auth Store (Pinia) | 단위 | 중간 |
| Company List/Filter UI | E2E | 낮음 |

### 테스트 도구

- **백엔드:** pytest + httpx (FastAPI TestClient)
- **프론트엔드:** Vitest (단위), Playwright (E2E)

## Out of Scope

- 기업 추천 기능 (방문자 행동 데이터 기반) — 사용자 데이터 축적 후 추후 고려
- 방문자 회원가입/마이페이지 — 현재는 비로그인 열람만 지원
- 다국어 지원 (한국어 단일 지원)
- 결제/유료 플랜 기능
- 기업 간 메시지/문의 기능
- 모바일 앱 (웹 반응형으로 커버)
- VPS 배포 설정 — 로컬 개발 이후 별도 결정
- CI/CD 파이프라인 — 초기 단계에서는 불필요

## Further Notes

- **프로젝트 이름 미정** — 추후 결정 필요
- **Ollama 제약:** 배포 환경에서 GPU 없는 서버는 응답 속도 저하 가능. 추후 Google Gemini Flash 또는 Groq API로 전환 옵션 열어둘 것
- **관리자 계정:** 초기에는 DB 직접 삽입 또는 스크립트로 생성, 별도 관리자 가입 페이지 불필요
- **PostgreSQL:** 로컬 개발 시 `docker-compose.yml`로 단일 명령 실행 환경 구성
